General

top

Content-Types

The API provides different content types which can be used to retrieve different types of representation of a requested resource. The actual content types differ for each resource so the ones supported are documented in the appropriate section.

The only content type supported by each resource is application/json.

Common Query Parameters

Except the Resources collection all other collections can be queried/filtered by using the following query parameters. Each parameter is optional and can only be contain one value. Multiple paramters are combined to an AND query. A resource may specify further query parameters to control/filter output. Special query parameters are described in each section.

Parameter Example Description
expanded
 
expanded=true

If true a list of partially expanded resource items is returned. The expansion provides more detailed information up to a certain point. However, it does not necessarily provide all item information as one may get when request the resource item itself.

The default is false.
service
 
service=srv_sdlfj2
 All resources where the given service id does match.
category
 
category=cat_sadf2d
 All resources where the given category id does match.
offering
 
offering=off_fdoi23
 All resources where the given offering id does match.
feature
 
feature=foi_adf092e
 All resources where the given feature id does match.
procedure
 
procedure=pro_asd23
 All resources where the given procedure id does match.
observedProperty 
observedProperty=phe_sdjf210
 All resources where the given observed property id does match.
lang 
lang=de

Specifies the language the response data shall have. English shall be the fallback. The REST-API should consider the following order to detect a locale:

  • User-agent (e.g. header information)
  • locale parameter
A fallback language cannot be specified as it depends on the data of the service provider.
Note that the options have to be URL-encoded appropriately!

Paging

Resource collections are paged by default to restrict large data access (configurable by the API provider .. use size=MAX to get the maximum of a page). The query parameters offset and size can be used to define exactly that part of the collection to be returned by the server.

Considering the Web Linking (RFC5988) approach the exact location of a page within a collection can be retrieved from response header information.

Paging Example

GET /api/v1/stations?offset=4&size=2 HTTP/1.1
Host: sensorweb.demo.52north.org
Accept: application/json
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Link: </api/v1/stations?offset=4&size=2>; rel="self"
Link: </api/v1/stations?offset=0&size=2>; rel="first"; title="first page"
Link: </api/v1/stations?offset=2349&size=2>; rel="last"; title="last page"
Link: </api/v1/stations?offset=2&size=2>; rel="prev"; title="previous page"
Link: </api/v1/stations?offset=6&size=2>; rel="next"; title="next page"

Content omitted for brevity.

Fuzzy Search

The query parameter q allows simple search over all available timeseries parameters:

  • Category
  • Service
  • Offering
  • Feature
  • Phenomenon
  • Procedure

Method Path Content-Type Description
GET /api/v1/?q=my%20search application/json Lists all resource collections available.

Common Status Codes

A client shall expect the following HTTP status codes returned by the API.

Code When
200 Status OK. All went fine.
400 Status Bad Request. The request data cannot be read.
404 Status Not Found. The resource is not available.
500 Status Internal Server Error. Please contact the webadmin. In some cases an error message will be shown. But in general the server logs will help to trace the problem.

Stations

URLs

API calls are triggered by using the following URLs.

Method Path Content-Type Description
GET /api/v1/stations application/json Lists all measured stations available.
GET /api/v1/stations/{id} application/json Lists the measured stations with id {id}.

Query Parameters

The collection can be queried by optional filter parameters.

Parameter Example Description
within
[without CRS]		 
within={
  'll': { 
    "type": "Point", 
    "coordinates": [<lgt>,<lat>]
  },
  'ur': { 
    "type": "Point",
    "coordinates": [<lgt>,<lat>]
  } 
}

Stations within a given bounding box.

If you want a different CRS than the default (WGS84) you can add your own to the GeoJSON geometry, e.g. 'crs':{'type':'name','properties'{'name':'EPSG:25832'}}.

See the following example where to set the CRS attribute.
near
[without CRS]		 
near={
  "center": { 
    "type": "Point", 
    "coordinates": [<lgt>,<lat>]
  }, 
  "radius":"500" 
}

Stations around a given point. The radius unit is in kilometers (km).

If you want a different CRS than the default (WGS84) you can add your own to the GeoJSON geometry, e.g. 'crs':{'type':'name','properties'{'name':'EPSG:25832'}}.

See the following example where to set the CRS attribute.
near
[with CRS]		 
near={
  "center": { 
    "type": "Point", 
    "coordinates": [<lgt>,<lat>],
    "crs": {
      "type": "name",
      "properties": {
        "name": "EPSG:25832"
      }
    }
  }, 
  "radius":"500" 
}

Stations around a given point. The radius unit is in kilometers (km). The point's coordinates relate to CRS 'EPSG:25832' (ETRS89 / UTM zone 32N).

You may be interested in the official EPSG database or other links you can resolve spatial reference systems by code.
For this resource further common query parameters can be used.
Note that the options have to be URL-encoded appropriately!

Timeseries

top

URLs

API calls are triggered by using the following URLs.

Method Path Content-Type Description
GET /api/v1/timeseries application/json Lists all timeseries available.
GET /api/v1/timeseries/{id} application/json

Lists the metadata of timeseries with id {id}.
GET /api/v1/timeseries/{id}/data application/json

Lists the raw data of timeseries with id {id}.
GET /api/v1/timeseries/{id}/data/{interval} image/png

Returns an image containing a pre-rendered graph for timeseries with id {id}. Supported interval values are the following:

  • lastDay
  • lastWeek
  • lastMonth
POST /api/v1/timeseries application/json Request the metadata for multiple timeseries. The requested timeseries ids have to be contained in the POST request.
document request POST structure.
POST /api/v1/timeseries application/pdf
application/vnd.ms-excel
text/csv 		Exports the given stack of timeseries. The requested timeseries ids have to be contained in the POST request. Note that Excel and CSV export are optional.
document request POST structure.
POST /api/v1/timeseries/{id}/data image/png

Renders the data of timeseries with id {id}. You can control the rendered output (color, line width, etc) by passing style options.

Document query parameter style. Also available refValues must be stylable.

Query Parameters

Timeseries can be queried with the following filter parameters.

Parameter Example Description
timespan 
timespan=PT12H/2013-08-06
 An ISO8601 formatted period. Only backslash formatted values are supported.
style 
style={
  "color": "#2f2f2f",
  "diagram": {
    "type": "graph",
    "properties": {
      "lineType": "solid",
      "lineWidth": 2
    }
  }
}

Style options how the timeseries shall be rendered if an image/png shall be the output.

A more detailed explanation about styling can be found under the styles section.
generalize 
generalize=true
 Indicates that timeseries data shall be generalized. Default is false.
